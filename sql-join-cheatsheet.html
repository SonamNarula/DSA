01 SQL JOIN Decision Tree

Click on any decision node below to expand the technical explanation and view syntax templates instantly.

Read the Question carefully
Analyze the requirements & schema. Start here.
Do you need only matching rows?
Are unmatched records completely filtered out?
Path A: Yes
If you only need rows where keys exist in both tables (intersection), use an INNER JOIN.

SELECT * FROM A INNER JOIN B ON A.key = B.key;
Do you need all rows from the Left table?
Keep table A intact while right table B is optional?
Path B: Yes
Use a LEFT JOIN. This preserves the left table and pads unmatched right rows with NULLs.

SELECT * FROM A LEFT JOIN B ON A.key = B.key;
Do you need all rows from the Right table?
Keep B intact while left table A is optional?
Path C: Yes
Use a RIGHT JOIN. Alternatively, swap table order and use a LEFT JOIN (preferred for style).

SELECT * FROM A RIGHT JOIN B ON A.key = B.key;
Do you need every row from both tables?
Union of both left and right unmatched rows?
Path D: Yes
Use a FULL OUTER JOIN. Matches where possible, otherwise pads missing fields with NULLs.

SELECT * FROM A FULL OUTER JOIN B ON A.key = B.key;

04 LEFT JOIN vs INNER JOIN
Metric INNER JOIN LEFT OUTER JOIN
Core Objective Return only rows that match the join condition in BOTH tables. Return ALL rows from the left table, plus matching rows from the right.
Output Difference Rows without matches in either table are completely filtered out. Left table rows are always kept. Unmatched right side is filled with NULLs.
NULL Handling Omit NULL key comparisons (they do not match). Includes records with NULL keys on the left table.
Typical Traps Accidentally dropping critical records (e.g. users without orders). Assuming left joins are always faster. Turning it into an inner join in the WHERE clause.
05 LEFT JOIN vs RIGHT JOIN

A RIGHT JOIN is logically equivalent to a LEFT JOIN with the tables swapped.

TableA RIGHT JOIN TableB ON TableA.id = TableB.id
≡
TableB LEFT JOIN TableA ON TableB.id = TableA.id

Style Preference: In professional environments, developers almost universally prefer LEFT JOIN over RIGHT JOIN because SQL reads left-to-right. Placing the primary table on the left aligns the query with natural reading order.

07 ON vs WHERE Filter Placement

Placing a query filter in the ON clause vs the WHERE clause behaves identically for INNER JOIN, but causes drastically different outcomes for outer joins (LEFT, RIGHT, FULL).

Condition in ON (LEFT JOIN)

Filters the right table before/during the join phase. The output keeps all left rows intact, setting right columns to NULL if unmatched. 

SELECT e.name, d.name
FROM employees e
LEFT JOIN departments d ON e.department_id = d.id
  AND d.status = 'Active'; -- Keeps all employees!
Condition in WHERE (LEFT JOIN)

Filters the result set after the join phase. Rows where department details are NULL are dropped because NULL = 'Active' is UNKNOWN. 

SELECT e.name, d.name
FROM employees e
LEFT JOIN departments d ON e.department_id = d.id
WHERE d.status = 'Active'; -- Converts to INNER JOIN!

FROM table_a a
INNER JOIN table_b b ON a.key = b.key;

FROM table_a a

SELECT e.name, m.name AS manager
FROM employees e
LEFT JOIN employees m ON e.manager_id = m.id;

SELECT *
FROM employees

FROM employees
NATURAL JOIN departments;

13 Memory Tricks & Mnemonics

Need matching only?

Remember: INNER = INTERSECTION. Think of overlapping circles where only rows with coordinates in both sets are preserved.

Need everyone?

Remember: LEFT preserves the LEFT. If the instruction says "keep all users", place the "users" table first in your query configuration.
